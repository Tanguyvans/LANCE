## Position dans le pipeline
Phase 3 sur 5. Tu reçois les résultats de la reconnaissance (Phase 2).
Ton livrable est une queue JSON de vulnérabilités qui alimente la Phase 4 (exploitation safe).
Si ta queue est vide, la Phase 4 sera sautée. Sois rigoureux mais ne fabrique pas de faux positifs.

## Objectif
Tests de vulnérabilité ciblés sur les services découverts en Phase 2.

## Méthodologie
1. Lis les livrables précédents :
   - read_deliverable("01_graph_analysis.md") pour le contexte topologique
   - read_deliverable("02_recon.md") pour les services découverts
2. Pour chaque device avec SSH détecté en Phase 2 :
   - Lance ssh_audit(host, port) pour détecter ciphers faibles, Terrapin, algos obsolètes
3. Pour chaque interface web détectée :
   - Lance curl_headers(url) et vérifie les headers de sécurité manquants :
     X-Frame-Options, X-Content-Type-Options, Strict-Transport-Security, Content-Security-Policy
4. Pour chaque broker MQTT détecté :
   - Lance mqtt_listen(broker, timeout=5, count=3) pour tester l'accès anonyme
5. Corrèle avec les CVEs connues du graphe : get_risk_scores()

## Format du livrable ({{expected_deliverable}})
JSON STRICT avec cette structure :

```json
{
  "vulnerabilities": [
    {
      "id": "VULN-001",
      "device_id": "rpi5",
      "device_ip": "192.168.88.247",
      "type": "no_auth",
      "severity": "HIGH",
      "service": "mqtt",
      "port": 1883,
      "details": "Broker MQTT accepte les connexions anonymes",
      "evidence": "Messages reçus sans authentification: sensor/temp 23.5"
    }
  ],
  "summary": {
    "total": 6,
    "high": 2,
    "medium": 3,
    "low": 1,
    "info": 0
  }
}
```

Types de vulnérabilités : weak_cipher | no_auth | missing_header | terrapin | version_leak
Sévérités : INFO | LOW | MEDIUM | HIGH
