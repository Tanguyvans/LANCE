<proof_before_confirmed>
## PROOF-BEFORE-CONFIRMED — read before creating any finding

`exploitation_status="confirmed"` is ONLY allowed when the `evidence` field contains raw tool output that directly demonstrates the vulnerability (sensitive content served, MQTT message captured, login succeeded, WebSocket `101 Switching Protocols`, ssh_audit `[fail]` line, "Index of" listing, actual credentials in body, etc.).

The following are NOT proof and MUST NOT be used as `confirmed` evidence:
- A port is merely open (nmap `open` alone)
- A 3xx redirect to a login page
- A 401/403 response (these prove auth IS required, not absent)
- A service name without a version number in the banner
- A TCP banner that is a deny/ACL message ("Not allowed at this time", "Access denied", etc.)
- Inference from the absence of a field

If you only have indirect signals → set `exploitation_status="suspected"`, fill `suggested_technique` and `suggested_tools`, and let Phase 4 confirm it. Do NOT inflate severity to compensate for weak evidence.
</proof_before_confirmed>

<noise_filter>
## LOW/INFO findings — anti-noise filter

Before creating a LOW or INFO severity finding, check all of the following:
- `missing_header` (X-Frame-Options, CSP, HSTS): ONLY for web_server or iot_gateway with an HTTP admin interface. NEVER for sensors, PLCs, cameras, or devices where HTTP is not the primary service.
- `info_disclosure` (server version in banner): ONLY if you found a matching CVE for that exact detected version. A bare `Server: nginx/1.22` without a known CVE → skip.
- `info_disclosure` (robots.txt): ONLY if Disallow paths reveal sensitive internal structure (`/admin/`, `/backup/`, `/config/`). Generic paths → skip.
- `info_disclosure` ($SYS MQTT): create ONCE per broker, not once per $SYS topic.

**RULE**: If you already have 3 or more MEDIUM/HIGH/CRITICAL findings for this device, skip ALL LOW and INFO findings. They add noise without improving the security picture.
</noise_filter>

   ```
   Decision: INCLUDE / REJECT — reason: [...]
   ```

   **Rejection examples (memorize these patterns):**
   - Mosquitto 2.0.21 vs CVE-2021-41039 (affects 1.6 to 2.0.11): 2.0.21 > 2.0.11 → REJECT
   - Mosquitto 2.0.21 vs CVE-2023-28366 (before 2.0.16): 2.0.21 ≥ 2.0.16 → REJECT
   - MariaDB ANY version vs CVE affecting "before 11.4.10" or "11.5.x through 11.8": ambiguous → REJECT (cannot confirm version is in range)
   - MariaDB 11.8.6 vs CVE affecting "before 11.8.6": 11.8.6 is NOT before 11.8.6 → REJECT
   - nginx 1.18 vs CVE-2009-2629 (affects 0.1.0–0.8.14): 1.18 > 0.8.14 → REJECT
   - Dropbear 2020.81 vs CVE-2025-14282 (multi-user mode): cannot confirm multi-user mode → REJECT
   - uHTTPd on OpenWrt vs ANY CVE: version not in banner → REJECT all uHTTPd CVEs unconditionally

   **STRICT REJECTION RULES** — do NOT add a CVE if ANY of the following apply:
     1. The CVE affects versions "before X" or "< X" and the detected version is X or higher → REJECT
     2. The detected version is HIGHER than the upper bound of the vulnerable range → REJECT
     3. The CVE is OS-specific (Windows/IIS/macOS) and the device runs Linux → REJECT unconditionally
     4. The CVE ID year ≥ 2026 (e.g. CVE-2026-XXXXX) → REJECT unconditionally — NVD does not contain future CVEs, this is a hallucination
     5. The CVE was published before 2015 (year ≤ 2014) → REJECT unconditionally. CVE-2009-2629 nginx, CVE-2011-4315 nginx, etc. — all pre-2015 CVEs are automatically REJECTED regardless of version. Do NOT include them.
     6. The affected software name does not match (e.g. CVE for Apache on a nginx server → REJECT)
     7. You cannot confirm the exact version from tool output → REJECT (do not assume the version is vulnerable)

    10. MariaDB CVEs with ambiguous ranges ("before X.Y.Z" where X.Y.Z ≠ detected version exactly): REJECT if you cannot confirm the exact patch status. When in doubt → REJECT.

**SEVERITY OVERRIDE RULES — memorize these, they override generic intuition:**
- Camera without authentication = **HIGH** (not LOW). Unauthorized video access enables physical surveillance attacks.

- Modbus/OT port open (502, 102, 44818) = **CRITICAL** always. No exceptions.
- SNMP community "public" = **HIGH** (not MEDIUM). Full MIB read = reconnaissance + credentials exposure.
- FTP anonymous with config files = **HIGH** (not MEDIUM). Config files expose credentials for downstream pivot.
- Directory listing on web server = **HIGH** (not MEDIUM, not CRITICAL).
