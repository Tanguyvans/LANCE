<methodology>
## Methodology — MANDATORY for each service on the device

1. Read context from previous phases (ONCE, at the start — do NOT re-read these later):
   - read_deliverable("01_graph_analysis.md") for the topology
   - read_deliverable("02_recon.md") for discovered services
   - search_history("{{device_ip}}", device_id="{{device_id}}") — check previous run findings for this device. If a finding is already confirmed with evidence, you may include it directly without re-running the test.
   RULE: Call each read_deliverable EXACTLY ONCE. Never call the same filename twice.

2. For each service on {{device_id}}, apply the appropriate tests:

   **SSH** (if detected):
   - ALWAYS call ssh_audit("{{device_ip}}") independently. Do NOT rely on 02_recon.md summaries — the recon agent may have misclassified results. Use 02_recon.md as context only.
   - Look for: weak ciphers (`[fail]` lines), Terrapin (CVE-2023-48795), obsolete algorithms
   - If Terrapin (CVE-2023-48795) is found: create type=terrapin finding. Do NOT also create a type=known_cve finding for the same CVE — it would be a duplicate.

   **HTTP/HTTPS** (if detected):
   - curl_headers("http://{{device_ip}}") — check security headers and server banner
   - MANDATORY path enumeration — call each one individually, do NOT skip:
     curl_headers("http://{{device_ip}}/backup/")
     curl_headers("http://{{device_ip}}/config/")
     curl_headers("http://{{device_ip}}/admin")
     curl_headers("http://{{device_ip}}/logs/")
     curl_headers("http://{{device_ip}}/firmware/")
     curl_headers("http://{{device_ip}}/api/devices")
     curl_headers("http://{{device_ip}}/api/status")
     curl_headers("http://{{device_ip}}/update")
   - For routers with uHTTPd/OpenWrt: additionally test curl_headers("http://{{device_ip}}/cgi-bin/luci"). For router role: if port 80 is open AND responds with ANY HTTP code (200, 301, 302, 403) → type=no_auth, severity=CRITICAL ("Router admin interface accessible from network — no WAN restriction"). The 403 response still confirms the service is exposed. Do NOT downgrade to HIGH for routers.
   - If return_code=7 (connection refused): still document the attempt, then skip remaining paths for this device
   - If stdout contains "Index of" or lists files → type=directory_listing, severity=HIGH
   - If stdout contains passwords, keys, or tokens → type=data_exposure, severity=CRITICAL. IMPORTANT: if MULTIPLE paths on the same device expose credentials (e.g. /backup/ AND /config/ both have sensitive data), create ONE single data_exposure finding and list all affected paths in evidence. Do NOT create one finding per path.
   - If /admin or /api/* returns 200 without auth → type=no_auth, severity=HIGH
   - If /update returns 200 (POST or GET accepted without auth) → type=insecure_update, severity=MEDIUM ("Firmware update endpoint accessible without authentication")
   - For iot_gateway role: if /firmware/ shows directory listing AND no .sha256 or .sig file is listed → add to insecure_update finding or create one: "OTA update endpoint unauthenticated and firmware has no integrity verification". Do NOT create a separate directory_listing finding for /firmware/ on iot_gateway — it is already captured by the insecure_update finding.

   **MQTT** (port 1883, or if device role is mqtt_broker):
   - ALWAYS call mqtt_listen("{{device_ip}}", topic="#", count=3, timeout=5) — do NOT skip even if 02_recon.md already mentions anonymous access. You MUST run the tool yourself.
   - CRITICAL exit code interpretation: return_code=27 (timeout) OR return_code=0 → anonymous access CONFIRMED → type=no_auth, severity=HIGH
   - Only return_code=5 means authentication is required (no finding)
   - If anonymous access is confirmed: you MUST create TWO separate findings:
     1. type=no_auth, severity=HIGH — "MQTT broker allows anonymous connections"
     2. Examine message content carefully. If ANY message contains passwords, tokens, API keys, credentials, or secret-like data (even partial) → create a SECOND finding type=data_exposure, severity=MEDIUM. Quote the actual message content as evidence. Do NOT merge this into the no_auth finding.
   - For mqtt_broker role: the minimum expected output is 2 findings (no_auth + data_exposure) if any credential-like data appears in topics.

   **Telnet** (for routers and gateways — device type=router or gateway):
   - ALWAYS run nmap_scan("{{device_ip}}", ports="23") to check for Telnet. This is MANDATORY even if 02_recon.md does not mention Telnet.
   - If port 23 is open → type=insecure_protocol, severity=MEDIUM (Telnet exposes credentials in cleartext)
   - severity is MEDIUM (not HIGH) for insecure_protocol/Telnet.

   **Database** (for db_server role — MANDATORY):
   - ALWAYS run nmap_scan("{{device_ip}}", ports="3306,5432,27017", skip_discovery=true)
   - If port 3306 is open: run nmap_scan("{{device_ip}}", ports="3306", scripts="mysql-empty-password", skip_discovery=true)
     - If script confirms empty root password → type=default_credentials, severity=CRITICAL
     - If port is merely open (no auth confirmed) → type=network_exposure, severity=HIGH ("Database port exposed to network without firewall")
   - If port 5432 is open → type=network_exposure, severity=HIGH

   **Default credentials** (for SSH — ssh_server role ONLY):
   - ONLY for devices with role=ssh_server: run nmap_scan("{{device_ip}}", ports="22", scripts="ssh-auth-methods", skip_discovery=true).
   - For iot_gateway, router, or any other role: do NOT run ssh-auth-methods and do NOT create a default_credentials finding from SSH password auth detection. SSH password auth is expected on embedded devices.
   - Read the nmap output carefully. If the output contains "password" anywhere in the supported auth methods (e.g. "publickey, password" or "password" alone) → password authentication IS enabled → MANDATORY finding:
     type=default_credentials, severity=HIGH
     details: "SSH password authentication enabled — default IoT credentials (admin:admin, root:root) must be tested manually"
     evidence: paste the exact nmap ssh-auth-methods output line
   - "publickey, password" counts — password being one of several methods is still a finding.
   - Do NOT skip this step for ssh_server role even if ssh_audit showed no issues.

   **Custom ports**:
   - nmap_scan("{{device_ip}}", ports="port1,port2")

3. **CVE LOOKUP — MANDATORY**:
   For EACH service with a known version, run cve_search(query):
   - Build a CPE or keyword query from the service name/version
   - Examples: cve_search("cpe:2.3:a:eclipse:mosquitto:2.0.11:*:*:*:*:*:*:*"), cve_search("OpenSSH 8.9"), cve_search("nginx 1.18.0")
   - cve_search checks the local cache first (fast), then queries NVD live on cache miss. Prefer it over nvd_lookup.
   - RULE: Use the software name and version as detected by nmap, NOT an assumed software.

   **MANDATORY VERSION CHECK — for EVERY CVE candidate, write this out before including it:**
   ```
   Detected version: [X.Y.Z]
   CVE range: [exact wording from CVE description]
   Is [X.Y.Z] inside the vulnerable range? YES/NO
   Decision: INCLUDE / REJECT — reason: [...]
   ```

   **Rejection examples (memorize these patterns):**
   - Mosquitto 2.0.21 vs CVE-2021-41039 (affects 1.6 to 2.0.11): 2.0.21 > 2.0.11 → REJECT
   - Mosquitto 2.0.21 vs CVE-2023-28366 (before 2.0.16): 2.0.21 ≥ 2.0.16 → REJECT
   - MariaDB 11.8.6 vs CVE affecting "before 11.4.10, 11.5.x through 11.8": ambiguous range — if 11.8.x is the upper boundary and 11.8.6 could be patched → REJECT (version not confirmed as unpatched)
   - MariaDB 11.8.6 vs CVE affecting "before 11.8.6": 11.8.6 is NOT before 11.8.6 → REJECT
   - nginx 1.18 vs CVE-2009-2629 (affects 0.1.0–0.8.14): 1.18 > 0.8.14 → REJECT

   **STRICT REJECTION RULES** — do NOT add a CVE if ANY of the following apply:
     1. The CVE affects versions "before X" or "< X" and the detected version is X or higher → REJECT
     2. The detected version is HIGHER than the upper bound of the vulnerable range → REJECT
     3. The CVE is OS-specific (Windows/IIS/macOS) and the device runs Linux → REJECT unconditionally
     4. The CVE ID year is the current year or later (e.g. CVE-2026-XXXXX) → REJECT unconditionally — NVD does not contain future CVEs, this is a hallucination
     5. The CVE was published before 2015 → REJECT unless the installed version is confirmed to be in the exact vulnerable range (e.g. nginx 0.x, not 1.x)
     6. The affected software name does not match (e.g. CVE for Apache on a nginx server → REJECT)
     7. You cannot confirm the exact version from tool output → REJECT (do not assume the version is vulnerable)
     8. uHTTPd/OpenWrt: if the banner shows only "uhttpd" without a version number → REJECT all CVEs for uHTTPd (version unconfirmed)
   - Only include CVEs where the detected version falls INSIDE the vulnerable range. If unsure, do NOT include.
   - Do NOT skip this step, even if get_risk_scores() already has CVEs

4. Cross-reference with graph CVEs: get_risk_scores() and get_device_info("{{device_id}}")
</methodology>
